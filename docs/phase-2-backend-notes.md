# Phase-2 backend design notes

Brain dump from the niri pathfinding. Captures everything the new compositor's color-management subsystem needs to express cleanly that niri+smithay could only express awkwardly. Not an implementation plan; an inventory of architectural primitives and decisions.

Sister docs in this repo: [phase-2-reuse-map.md](phase-2-reuse-map.md) (what we depend on / copy / replace), [phase-2-progress.md](phase-2-progress.md) (what's built so far).

## The pipeline that has to fit cleanly

Every reference compositor doing real color management converges on this shape:

```
per-client surface
   └─ decode shader (TF + primary matrix, knows the client's color description)
        └─ writes into intermediate (linear-light, fp16, defined color space)
            └─ blend / composite (in linear)
                └─ postprocess shader (per-output: tone-map + primary matrix + OETF)
                    └─ scanout buffer (output's native encoding: sRGB or PQ)
                        └─ KMS modeset properties (Colorspace, HDR_OUTPUT_METADATA)
                            └─ panel
```

Two stages of color transform: **per-element decode at draw time**, and **per-output encode at scanout time**. Compositing happens in linear-light in between. This is non-negotiable for correct alpha blending and any per-client color-space handling.

The phase-2 renderer must make both transforms first-class: a `RenderElement` (or whatever we call it) must carry its source color description and run the appropriate decode shader at draw time; an output must carry its target color description and run the appropriate postprocess at scanout time. Neither can be bolted on after the fact.

## The impedance mismatch with smithay

What made this hard in niri:

- **Smithay's `GlesRenderer` and the `RenderElement::draw` API assume one default texture shader** that does texture sampling + matrix + alpha. There's no per-element decode hook. To do per-element decode, every element type that draws a texture has to be replaced or wrapped with one that uses a custom shader. That's `WaylandSurfaceRenderElement`, `LayerSurfaceRenderElement`, `MemoryRenderBufferRenderElement`, `SolidColorRenderElement`, ... and they're not niri's types, they're smithay's. We'd be rewriting or wrapping a substantial chunk of smithay's element ecosystem.
- **`DrmCompositor::render_frame` owns the scanout buffer** and binds it as the render target itself. We had to side-step it for the HDR fork by rendering into an intermediate via a separate code path, then submitting a single fullscreen `ShaderRenderElement` to `render_frame`. Works, but the architecture wants "render into output buffer directly" while we wanted "render into intermediate, postprocess, then submit." The shape is wrong.
- **Niri's `OutputRenderElements<R>` enum** is a closed sum type generated by a macro, with `R: NiriRenderer`. Adding a new variant for a color-aware texture element means a per-variant impl in the macro expansion. Workable but not what you'd design.
- **`MultiRenderer` and `as_gles_renderer()`** complicate the offscreen render path. We worked around it by routing the HDR intermediate through the primary GPU's `GlesRenderer` directly, which makes multi-GPU client buffers an uncertain edge case for HDR outputs.

None of these are smithay defects — they're consequences of smithay being designed for SDR compositors where one texture shader is enough and direct scanout is the optimization. Color management is a different architectural axis than what smithay's renderer was built around.

For phase 2: the renderer abstraction should make a `RenderElement` (or whatever we call it) carry a color description as a first-class property, and the renderer pipeline should be "decode at draw, accumulate in intermediate, postprocess to scanout" by default — even for SDR-only outputs. The "direct scanout" optimization is a *specialization* of the general path (when client's description matches output's, skip the round-trip), not a separate path.

## Decisions already made (from phase-1 design discussion)

| Decision | Choice | Why |
|---|---|---|
| Intermediate color space | **BT.2020 primaries, absolute-nits linear, fp16** (1.0 = 1 nit) | BT.2020 covers all panels we care about. Absolute nits aligns with PQ semantics, lets us reason in real luminance numbers throughout, matches gamescope's proven approach for HDR. fp16 is the only format with enough range+precision for the absolute-nits encoding. |
| Per-element decode mechanism | **Single shader with uniform-driven branches** (one decode shader, parameterized by source TF + primary matrix + ref luminance) | Simpler initially; KWin's per-(TF, primary) variant approach is an optimization we can adopt later if profiling shows it's needed. Avoids shader-cache complexity in the first cut. |
| Per-output OETF | **Pluggable per output**: sRGB-encode for SDR, PQ-encode for HDR, identity/other for future cases | Each output declares its target color description; the postprocess shader's encode stage is selected from that. |
| Pipeline unification | **Every output goes through the intermediate + postprocess**, even SDR-only | Eliminates the SDR/HDR fork. Adds one render pass per frame to SDR outputs (negligible at 1080p; bounded at 4K@60). |
| Frog protocol | **Defer** to after basic `wp_color_management_v1` works | Frog is the de-facto standard for HDR games today; worth supporting eventually for OLED + Steam. But phase-1 spec-aligned protocol first. |

## What needs to be first-class state

**Per-surface state** (lives with the Wayland surface, set by clients via `wp_color_management_v1`):

- Transfer function: enum {sRGB, BT.1886, PQ (SMPTE ST 2084), HLG, linear, custom-parametric}
- Primaries: enum {sRGB/Rec.709, DCI-P3, BT.2020, Adobe RGB, custom-xy-coordinates}
- Reference white luminance (nits): SDR is 100 by convention, HDR clients can specify
- Optional: max content light level, max frame-average light level (CTA 861.G)
- Optional: mastering display metadata for HDR content
- Rendering intent: enum {perceptual, relative-colorimetric, saturation, absolute-colorimetric}

Default for clients that don't speak the protocol: sRGB transfer, sRGB primaries, 100-nit ref white, perceptual intent.

**Per-output state** (lives with the output, set by compositor config + EDID):

- Target color description (what to encode to before scanout): primaries + TF + peak luminance + ref white
- Tone-mapping curve choice (BT.2390 EETF default, configurable)
- Calibration matrix (3×3 in linear-light, derived from measurements)
- Calibration 1D LUT per channel (optional, for non-linear corrections)
- Calibration 3D LUT (optional, full ICC-style)
- Reference white nits in SDR-source-on-HDR-output mode (we hardcoded 200 in niri; should be per-output)
- HDR static metadata for the panel (max_cll, max_fall — what we put in `HDR_OUTPUT_METADATA`)

**Per-frame derived state** (computed each frame for each output):

- Currently-bound decode shader uniforms per element
- Currently-bound postprocess shader uniforms (tone map, OETF mode, ref luminance)
- Calibration application stage (post-decode, pre-OETF, in shader)

## KMS / amdgpu gotchas to bake in from the start

Not relearn-on-greenfield items:

1. **amdgpu has no `DEGAMMA_LUT` at CRTC level.** All linearization must happen in shaders. The new per-plane `COLOR_PIPELINE` (gen 2) exposes degamma per-plane on DCN3+ when amdgpu lights it up on stable kernels (not present in 7.0.x). Plan: shader degamma is the always-works path; per-plane hardware degamma is a future optimization for the cards that support it (Navi 21 DCN3 will; Vega 20 DCN1 won't).
2. **CRTC `CTM` is post-blend, post-shader, gamma-space.** Useful as a calibration trim for SDR outputs *if* coefficients are derived in gamma space (we did this — see `spyder/measurements/.../calibrate.py`). In HDR mode the CRTC sees PQ-encoded scanout — the gamma-space CTM produces wrong math. For phase 2, calibration matrix lives in the shader (linear-light, post-decode, pre-OETF). The CRTC CTM is then either unused or a final per-channel trim only.
3. **`HDR_OUTPUT_METADATA` change triggers a full modeset** in amdgpu (~165 ms on our hardware). Don't toggle HDR per frame; treat it as a per-output mode setting that's stable across config reloads.
4. **`max bpc` is a connector property**, not a plane property. Setting it requests deeper link bit depth at modeset; kernel clamps to EDID-advertised range silently.
5. **Legacy property writes for HDR_OUTPUT_METADATA + Colorspace work** through the kernel's atomic shim — no need for explicit `AtomicModeReq` plumbing unless you want it for other reasons.
6. **`Colorspace` enum on amdgpu**: `{Default, BT709_YCC, opRGB, BT2020_RGB, BT2020_YCC}`. No `DCI-P3_RGB_D65` or anything between. For HDR signaling we use `BT2020_RGB`. For SDR signaling we use `Default` and the panel decodes as sRGB / Rec.709.
7. **`max_cll` lying-low.** Per KWin's notes and our own panels (LU28R55), Samsung tone-mappers misbehave with honest peak values — they crush highlights unnecessarily. Lying the `max_cll` field lower than the true panel peak (e.g., advertise 250 when peak is 400) gives a more pleasant picture. Per-panel quirk; expose as config.
8. **fp16 (`Xbgr16161616f`) scanout works on both DCN1 (Vega 20) and DCN3 (Navi 21).** We negotiated XB4H on all 6 outputs. The smithay path "just worked" once we added `Xbgr16161616f` to the format preference list before `Xbgr2101010` and the 8-bit fallbacks.
9. **EDID HDR static metadata** is parseable and tells us per-panel peak luminance, native primaries, and supported EOTFs. We didn't wire EDID parsing in phase 1 (hardcoded the LU28R55 as HDR400-PQ); phase 2 should consume EDID to advertise per-output capabilities accurately.

## Lessons from reference compositors

| Compositor | Intermediate | Decode location | Notable |
|---|---|---|---|
| **KWin** | gamma 2.2 (NOT linear) | per-surface shader | Deliberately chose perf over correctness; 10-bit intermediate; relies on hardware OETF offload |
| **Mutter** | BT.2020 linear | per-surface decode in shader | Textbook "right thing"; pays fp16 cost; integrates DRM plane color properties |
| **gamescope** | scRGB linear, absolute nits | pre-decode per fullscreen game | Game-focused; extensive inverse-tone-mapping for SDR-on-HDR; built around fullscreen Steam workflow |
| **Hyprland** | gamma-ish, shader-driven | per-surface shader | Still pre-protocol; uses monitor-rule config for color mode selection |

We're closest to Mutter conceptually (BT.2020 linear intermediate, per-surface decode in shader), but we want gamescope's absolute-nits encoding because it keeps the math in real units throughout.

KWin's gamma-2.2 choice is a forced perf optimization for older hardware and high-update-rate displays. We may need to revisit this for the 240Hz OLED at 4K — but the right plan is "build correct first, measure, optimize where needed" not "design around the perf hack."

## Tone mapping

What we deferred in phase 1; the greenfield needs it from day 1:

- **HDR client on SDR output**: tone-map highlights down to SDR range (BT.2390 EETF is the standard reference; Reinhard / Hable for simpler approximations). Loses HDR-ness but keeps colors plausible.
- **SDR client on HDR output**: scale SDR ref white to a chosen nit level (we hardcoded 200; should be configurable per output). Optionally apply inverse tone-mapping to push highlights brighter (controversial; usually best left identity).
- **HDR client on HDR output**: if peak luminances match exactly, identity. Otherwise tone-map between them. The "panel can do 400 nits, content is mastered for 1000 nits" case requires highlight roll-off — gamescope and Mutter do this.
- **Cross-gamut**: clients in DCI-P3 on a BT.2020 panel, or sRGB clients on a wider-gamut panel. Each needs a primary conversion matrix and out-of-gamut handling. Hard-clip is simplest; soft-clip (perceptual) is the "right" thing but needs a 3D LUT or shader math.

For phase 2: tone mapping is a per-output config (curve choice + parameters), applied in the postprocess shader between the calibration matrix and the OETF.

## Calibration data flow

The spyder loop in `/home/christian/workspace/spyder/` is the calibration tool. In phase 1 it produced gamma-space CRTC CTMs (3×3 matrices, see `calibrate.py`). For phase 2:

- **What the spyder generates**: per-output measurements (R/G/B primaries chromaticity, white point, gamma curve, white luminance) at a reasonable patch grid (currently 16-bit grey ramp + RGB primaries; could extend to full 3D grid).
- **What the compositor consumes**: per-output 3×3 matrix (in linear-light, post-decode, pre-OETF), and optionally per-channel 1D LUT and/or 3D LUT. Format probably matches what ICC profiles encode — load .icc, parse matrix/TRC/cLUT, push to shader.
- **Where calibration lives in the pipeline**: between the postprocess's input (linear BT.2020 absolute nits) and the postprocess's OETF stage. Applied as `out_nits = M_cal * in_nits` (then per-channel 1D LUT, then 3D LUT, then OETF).
- **Color-aware calibration is the goal**: the spyder tool should be a `wp_color_management_v1` client itself, emit absolute-luminance / known-primary patches via the compositor's HDR path, measure with the colorimeter, derive corrections. This generates measurements against the *real* pipeline rather than a partially-broken stand-in.

## What we wish niri had

Things we'd build differently from day 1:

- **Color description as a first-class property of every render element**, not an opt-in metadata field. SDR clients carry "sRGB, 100 nits ref white" by default; HDR clients carry their declared description. The renderer dispatches the decode shader from this.
- **Renderer abstraction designed around "decode per element, postprocess per output"** rather than around "draw a texture, write it to a buffer." Maybe an explicit `DecodeStage`/`CompositeStage`/`EncodeStage` pipeline split, with each stage taking strongly-typed inputs.
- **Output's target color description as a config, not a code path**. SDR outputs are a degenerate case (target = sRGB primaries + sRGB TF + 100 nit ref) of the same machinery that handles HDR outputs.
- **EDID parsing for HDR static metadata block** from the start. We hardcoded panel specs in phase 1 (HDR400 LU28R55, 400 peak, etc.); phase 2 should consume EDID to set defaults that config can override.
- **Per-output calibration data path** built into the config schema, not retrofitted. Probably ICC profile load via `lcms2` (binding) or manual matrix/LUT parsing.

## Things deliberately punted

For phase 2 design, these are open:

- Renderer choice (Vulkan via `ash`, `vulkano`, `wgpu`, or pure FFI)
- KMS/DRM abstraction (reuse smithay's drm-rs-based layer, or rewrite — there isn't much there beyond ergonomic wrappers around `drm-rs`)
- Wayland protocol implementation (smithay's wayland-server crate is reusable independent of its renderer; or pure `wayland-server` from wayland-rs)
- Window management scope (stacking, tiling, none-just-fullscreen)
- Input handling (libinput direct, or smithay's wrapper)
- IPC mechanism (none, custom socket, niri-style varlink, dbus)
- Async runtime (calloop, tokio, custom)
- Animation framework (none if window management is minimal)
- Config format (KDL like niri, TOML, custom)

The color-management subsystem can be designed independently of most of these. The decisions that affect it: renderer (Vulkan vs GL changes the shader story), and protocol layer (smithay's wayland-server vs alternatives changes the surface-state hook story).

## Reusable artifacts from phase 1

- `hdr_pq.frag` — the PQ encoding shader. Math is correct, ports directly to Vulkan GLSL.
- `M_SRGB_TO_BT2020` matrix and the inverse `M_BT2020_TO_SRGB` — standard but documented in our shader with derivation context.
- `build_hdr_metadata_blob` in `tty.rs` — knows the byte layout of `struct hdr_output_metadata` (kernel-compatible 32 bytes including tail padding). Ports.
- `ResolvedHdrConfig` resolution logic — turns user-facing config into kernel-ready values. Ports.
- Spyder calibration pipeline (gamma-aware CTM derivation) — knows panel quirks. Will need a rewrite for absolute-luminance / linear-light targets, but the patch-measurement-loop infrastructure is sound.
- Domain knowledge: per-panel quirks (LU28R55 Samsung max_cll lying-low), kernel quirks (modetest -D pci: silently aliasing card0), modeset latency markers, the right format preference order.
