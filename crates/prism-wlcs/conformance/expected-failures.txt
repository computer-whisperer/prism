# ─── smithay-level subsurface synchronization gap ───────────────────────────
#
# Per the wl_subsurface spec, both `set_position` and `place_above`/
# `place_below` are *parent-committed* state: they take effect "whenever
# the state of the parent surface is applied", regardless of the
# subsurface's sync/desync mode. The pinned smithay
# (ff5fa7df392cecfba049ffed55cdaa4e98a8e7ef) instead ties `set_position`
# to the *subsurface's own* cached-state commit (sync child → applied on
# parent commit via commit_sync_surface_tree; desync child → applied on
# the subsurface's own commit) and applies PlaceAbove/PlaceBelow
# immediately. This is in core `PrivateSurfaceData::commit`
# (src/wayland/compositor/tree.rs), so smithay's own wlcs_anvil would fail
# these too — it is not a prism-specific bug.
#
# Real-world impact is low: desync subsurface clients (GTK4, Firefox,
# Mesa) commit the subsurface themselves alongside content, and position
# is normally set once at creation paired with a parent commit. These
# tests exercise the synthetic "position-only, exactly one side
# committed" orderings.
#
# Fixing requires either patching/forking the pinned smithay or
# reimplementing subsurface position + stacking with parent-commit
# semantics in prism (bypassing SubsurfaceCachedState). Deferred by
# decision 2026-05-24.
